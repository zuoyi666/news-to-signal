# News-to-Signal

This repository is the upstream signal-construction layer in the research stack. It builds standardized cross-sectional news signals and exports them for downstream consumption by the implementation-aware backtesting repository [cost-aware-backtest](https://github.com/zuoyi666/cost-aware-backtest).

## Problem

Can textual features from financial news predict short-horizon cross-sectional equity returns?

This project tests the hypothesis that:

- Sentiment and uncertainty features from headlines are informative
- Event-related language improves alpha signal quality
- Cross-sectional standardization allows stock-ranking signals usable in long-short testing

## Why it matters

Market microstructure and sentiment-driven repricing are often short-lived. A standardized research pipeline enables:

- Reproducible signal experiments
- Comparable baselines across runs
- Faster extension to other data sources and models

## End-to-End Workflow

This repository is designed to pair directly with [cost-aware-backtest](https://github.com/zuoyi666/cost-aware-backtest) in a two-repo research stack:

1. Research and construct standardized signals in `news-to-signal`.
2. Export a standardized run artifact under `artifacts/runs/exp_YYYYMMDD_name/`.
3. Consume that artifact in `cost-aware-backtest` for cost-aware validation under explicit execution assumptions.

## Data

- Reference files:
  - `data/reference/universe.csv` (original 50)
  - `data/reference/sp500_universe.csv` (S&P 500)
  - `data/reference/russell1000_universe.csv` (expanded universe)
  - `data/reference/loughran_mcdonald.csv` (sentiment dictionary)
- Raw and processed datasets are configured in `configs/data.yaml` and can be regenerated by pipeline runs.
- Large external datasets are placed under `data/external/` (gitignored unless explicitly tracked).

## Feature / Label construction

### Label

- Forward return label: `future_return_5d = log(P[t+5]/P[t+1])`
- Trade date is normalized per news date

### Features

- Sentiment score from Loughran-McDonald dictionary
- Uncertainty score from uncertainty dictionary words
- Event intensity from keyword dictionary
- Cross-sectional signals:
  - `signal_sentiment_only`
  - `signal_sentiment_minus_uncertainty`
  - `signal_full`

## Validation

Validation modules include:

- Cross-sectional IC and spread metrics
- Baseline comparison across candidate signals
- Walk-forward split and robustness checks

All experiments are standardized by run name and persisted as machine-readable artifacts.

## Downstream Usage

This project exports downstream-ready signal artifacts for implementation-aware validation in [cost-aware-backtest](https://github.com/zuoyi666/cost-aware-backtest).

Exported signal outputs:

- `signals.parquet` as the preferred signal delivery format
- `signals.csv` as a fallback export when parquet support is unavailable
- `signals_manifest.json` with file and schema metadata for downstream loaders

Standardized downstream signal schema:

- `timestamp`
- `asset`
- `signal_value`
- `signal_name`
- `signal_version`
- `horizon`
- `source`

Export location:

- `artifacts/runs/exp_YYYYMMDD_name/`

Recommended downstream consumption pattern:

- Prefer passing the completed run directory to downstream loaders.
- If a loader supports manifest-based discovery, pass `signals_manifest.json`.
- If needed, downstream projects may read `signals.parquet` directly and use `signals.csv` only as a compatibility fallback.

## Downstream Signal Export Contract

Each experiment exports a downstream-ready signal file under `artifacts/runs/exp_YYYYMMDD_name/`:

- `signals.parquet`
- `signals.csv` when parquet support is unavailable
- `signals_manifest.json`

Required exported columns:

- `timestamp`
- `asset`
- `signal_value`
- `signal_name`
- `signal_version`
- `horizon`
- `source`

Contract notes:

- `timestamp` is exported as a timezone-naive datetime.
- `asset`, `signal_name`, `signal_version`, `horizon`, and `source` are exported as strings.
- `signal_value` must be numeric.
- Export validation fails fast on missing required columns, invalid required values, or duplicate downstream signal keys before downstream handoff.
- Export files are written under `artifacts/runs/exp_YYYYMMDD_name/`, with `signals.parquet` as the preferred format and `signals.csv` only as a compatibility fallback.

This contract is aligned with the downstream [cost-aware-backtest](https://github.com/zuoyi666/cost-aware-backtest) repository so completed run directories can be consumed directly by downstream loaders for implementation-aware validation.

## Data and Artifact Policy

Files that are reasonable to commit:

- source code under `src/`
- configs under `configs/`
- small reference files under `data/reference/`
- documentation such as `README.md`, `reports/README.md`, and `artifacts/runs/README.md`

Files that should stay local:

- generated run outputs under `artifacts/runs/<run_name>/`
- raw, interim, and external data under `data/raw/`, `data/interim/`, and `data/external/`
- local planning notes and scratch files

## Limitations

- Data collection is API-dependent (Finnhub/FMP may rate limit/free-tier limits)
- Coverage is universe/date dependent
- Figure generation is currently lightweight and does not yet persist all exploratory plots

## Reproducibility

Use config-first execution and always use a fresh run name:

```bash
cp .env.example .env
pip install -r requirements.txt
python cli.py run --experiment-name baseline --universe russell1000
```

Legacy compatible path:

```bash
python run_phase1.py --synthetic --universe sp500
```
